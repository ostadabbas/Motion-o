I'm working on MotionR1: adding motion reasoning to a video VLM (Open-o3-Video style) with explicit <motion> tags (direction, speed) and a motion_trajectory reward in GRPO.

We've fixed several issues and training is running, but I want a full pipeline review to catch anything subtle we might have missed.

**Current pipeline**
- SFT: STGR-SFT-subset-motion-v3.json (5.7k samples, 87.5% multi-frame). Model learns <think> with <obj><box>at<t></t>s and <motion>direction (speed: X)</motion>.
- GRPO: STGR-RL-subset.json (5.8k samples). No reasoning_process in data; model generates <think> from scratch. Rewards get GT via kwargs: key_items, key_frames, answer, task, image_size (trainer passes all input columns except prompt/completion).
- Motion reward (simplified): (1) Match each predicted <obj><box>at<t> claim to closest GT frame (Open-o3 style). (2) If >=2 matches span different GT frames, parse direction from <motion> text and compute GT direction from GT bboxes; direction_score = 1.0 if pred_direction == gt_direction else 0.0. (3) Speed score = ratio of pred vs GT displacement magnitude over time. (4) motion_reward = 0.5 * direction_score + 0.5 * speed_score. No smoothness/accel term.

**Things we already fixed**
- GRPO was using STGR-RL-subset (0% samples with >=2 key_frames). We made motion reward work with sparse GT by matching each prediction to closest GT frame (like thk_spatial_reward); GT key_items can have multiple frames even when key_frames lists one.
- Motion reward was 0 always; now it's non-zero (~0.007–0.01) in logs.
- We removed cosine-similarity direction in favor of simple word matching (parse "upward"/"leftward" etc. from <motion> text; compute GT direction from bboxes; exact string match).
